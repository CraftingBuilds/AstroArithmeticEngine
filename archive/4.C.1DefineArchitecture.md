### 4.C.1 Define the Layered Composition Architecture

The Composition Engine shall operate as a layered pipeline. The orchestration document `__main__` shall function only as the entry script and coordinator. Its responsibilities are limited to loading configuration, invoking layer modules in order, passing structured inputs forward, recording execution state, and assembling approved outputs. `__main__` shall not contain chart interpretation logic, prose composition logic, critique logic, or revision logic beyond routing based on returned pass/fail results.

The execution order shall be:

`input_normalization -> significance_ranking -> interpretation -> composition -> critique -> revision -> final_assembly`

Each layer must complete and validate its output before the next layer begins. Downstream layers must not run on invalid or incomplete upstream data unless an explicit recovery mode is invoked. Revision may re-enter the cycle at the appropriate layer without forcing unrelated outputs to be rebuilt. Final assembly must remain separate from drafting, critique, and revision.

Layer responsibilities are fixed as follows:
- `input_normalization` reads structured chart facts and produces normalized internal analysis objects
- `significance_ranking` determines which chart factors matter most for the active section
- `interpretation` converts ranked chart factors into chart-specific analysis notes
- `composition` turns interpretation notes into section prose in the approved voice
- `critique` scores output against rubric rules
- `revision` rewrites failed section drafts based on critique findings
- `final_assembly` collects approved section outputs into the larger report structure

No layer may silently absorb another layer’s responsibilities.

All layers must use explicit structured inputs and outputs. `__main__` shall invoke only layer scripts or modules, receive explicit structured returns from each layer, record execution state for each section and each layer, stop safely when a required layer fails, and flag sections for review instead of forcing false success states.

Every layer shall define a formal input schema and a formal output schema. Every schema must include:
- `section_id`
- `chart_id`
- traceable source references back to normalized chart data
- a status field such as `ok`, `warning`, or `failed`
- machine-readable metadata for debugging and auditability

Layer output that does not conform to its declared schema must be rejected.

Structured data handoff is required throughout the pipeline. Layers shall pass dictionaries, objects, or typed records instead of loose prose whenever possible. Freeform text shall be limited to drafted prose and human-readable logs. Machine-readable factor lists, scores, trace references, critique findings, and revision outcomes must be preserved across the cycle. No downstream layer may rely on hidden assumptions absent from structured input.

Each layer must validate required fields before processing. Optional but important missing data must generate structured warnings. Required missing or malformed data must generate structured errors. Failed layers must not pass corrupted output forward as if it were valid. The system must preserve failure logs and distinguish recoverable failures from hard-stop failures.

The engine must remain modular. Each layer shall exist as its own script or module with a single primary responsibility. Layers must be replaceable or updateable without rewriting the full engine. Section behavior must not be hard-coded into unrelated layers. Voice rules, rubric rules, and stylebook data should remain external where practical. The architecture must support future expansion into additional chart types and sections while preserving maintainability.
